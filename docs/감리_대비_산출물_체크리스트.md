# 응용시스템 분야 감리 대비 — 단계별 산출물 체크리스트 & 자동 점검 가이드

> 출처: 행정안전부/한국지능정보사회진흥원(NIA) **정보시스템 감리점검해설서 V3.0** (2008.3.3 공지)
> 사업유형: 시스템개발(SD) · 객체지향·컴포넌트기반 모델 / 응용시스템 분야
> 점검 단계: 요구분석(SD21-2) → 분석/설계(SD22-2) → 구현(SD23-2) → 시험활동(SD14-1) → 운영준비(SD15-1)

이 문서는 감리 제출 전 **"산출물별 작성기준 + 점검항목 + 검토항목"** 을 단계별로 정리한 마스터 체크리스트입니다.
각 항목은 해당 감리코드(예: `SD21-2-A1`)에 1:1로 추적 가능하도록 구성했습니다.

---

## 0. 감리 점검 구조 한눈에 보기

감리점검해설서의 점검 논리는 다음 계층으로 구성됩니다.

```
기본점검항목 (예: SD21-2-A "현행업무분석/해결방안")
   └─ 검토항목 (예: SD21-2-A1 "현행 업무영역 분석을 통해 문제점/해결방안이 모색되었는가?")
        └─ 세부 검토내용 (업무처리 개선필요사항, 문제원인·영향·해결방안 등)
             └─ 주요 검토대상 산출물 (비전기술서, 사용자요구사항정의서 등)
```

각 검토항목은 **감리관점/점검기준 매트릭스**(무결성·기능성·편의성·충족성·효율성·일관성·준거성·준수성·실현성·보안성·계획적정성·절차적정성)로 평가됩니다.

## 1단계 · 요구분석 (SD21-2)

> **개요**: 업무영역 분석과 사용자 요구사항을 도출/분석하고, 시스템 기능에 대한 유스케이스 모형 정의 및 분석 클래스 도출을 적정하게 수행하였는지 점검.

### 1.1 필수 산출물 및 작성기준

| # | 산출물 | 작성기준 (감리 통과를 위한 필수 포함 요소) |
|---|--------|---------------------------------------------|
| 1 | **비전기술서** | 시스템 상위 수준 요구사항, 사업 목표·범위, 제약사항 명시 |
| 2 | **사용자요구사항정의서** | 기능/비기능 요구사항을 명확·측정가능하게 정의, 이해관계자 확인 서명, 고유 요구사항 ID 부여 |
| 3 | **현행업무 문제점 분석문서** | 현행 업무처리 개선필요사항, 문제 원인·영향·해결방안, 시스템 구축 시 제약사항 |
| 4 | **유스케이스 모형 기술서** | 액터/유스케이스 식별, 시스템 간 상호작용, 요구사항 추적성 확보 |
| 5 | **개념적 클래스 모형 기술서** | 개념 수준 분석클래스, 클래스 간 관계·연산·메시지 흐름 |
| 6 | **사용자 인터페이스 프로토타이핑 계획서** | 대상 업무 선정, 프로토타이핑 일정·범위·평가계획 |

### 1.2 점검항목 / 검토항목 체크리스트

- [ ] **`SD21-2-A1`** 현행 업무영역 분석을 통해 문제점과 해결방안이 모색되었는가?
  - [ ] 업무처리 개선 필요사항 도출
  - [ ] 문제 원인·영향·해결방안 도출
  - [ ] 시스템 상위수준 요구사항과 해결방안 간 연관성
  - [ ] 시스템 구축 시 고려할 제약사항 도출
- [ ] **`SD21-2-B1`** 사용자 요구사항이 적절하게 도출되었는가?
  - [ ] 요구사항 도출 방법 및 절차
  - [ ] 이해관계자의 참여 및 확인
  - [ ] 기능/비기능 요구사항의 적정성 및 명확성
  - [ ] 타 시스템과의 연계 및 인터페이스
  - [ ] 사용자 활용성, 시스템 효과성
- [ ] **`SD21-2-C1`** 기능적 요구사항이 유스케이스 모형으로 충분히 표현·상세화되었는가?
  - [ ] 기능요구사항과의 충분성·추적성
  - [ ] 외부시스템·시스템 간 상호작용 식별
- [ ] **`SD21-2-D1`** 유스케이스로부터 개념적 수준 분석클래스가 충분히 도출되었는가?
  - [ ] 클래스 간 관계·연산·메시지 흐름
- [ ] **`SD21-2-E1`** 사용자 인터페이스 프로토타이핑 계획이 적정하게 수립되었는가?
  - [ ] 프로토타입 대상업무 선정 적정성
  - [ ] 일정·범위·평가계획 적정성

## 2단계 · 분석/설계 (SD22-2)

> **개요**: 요구분석 결과와 업무·사용자 요구사항 상세분석에 따라 유스케이스 모형 및 클래스를 충분히 정제하고, 구현 가능한 수준으로 설계하였는지 점검.

### 2.1 필수 산출물 및 작성기준

| # | 산출물 | 작성기준 |
|---|--------|----------|
| 1 | **업무상세분석서** | 주요 업무 식별·우선순위, 업무 수행자-업무 상호작용, 업무 흐름, 법제도 정비 등 위험요인 |
| 2 | **유스케이스 모형 기술서** | 유스케이스 정의·역할·액터 목록(누락 없음), 시나리오·이벤트 흐름(기본·예외) |
| 3 | **사용자 인터페이스 흐름도** | 화면 구성요소·메시지 도출, 업무흐름에 따른 화면 간 제어흐름 |
| 4 | **화면·보고서 정의서** | 공통/개별 화면·보고서 레이아웃, 양식 설계의 적정성·편의성·일관성 |
| 5 | **클래스 명세서** | 클래스 정의·속성/연산·역할 명세, 관계 명세, 구현언어·기술특징 반영, 추적성 |
| 6 | **내외부 시스템 연계 설계서** | 연계대상 시스템·데이터 식별, 연계 기술방식, 주기·방법 |
| 7 | **컴포넌트 설계서** | 클래스·인터페이스 도출, 지속성 설계, 개발 가능 수준 |
| 8 | **가변성 정의서** | 공통성·가변성 분석, 후보 컴포넌트와 업무 간 차이 분석 |
| 9 | **재사용 컴포넌트 선정기준서** | 재사용 컴포넌트 선정기준·후보 선정 적정성 |
| 10 | **소프트웨어 아키텍처 정의서** | SW 아키텍처-컴포넌트 아키텍처 일관성 |
| 11 | **컴포넌트 아키텍처 정의서** | 컴포넌트 구조화·관계 정의 |
| 12 | **시스템 보안정책 관련 문서** | 사용자별/그룹별/업무별 접근권한, 감사기능, 보안기술 적용 |
| 13 | **단위시험 계획서** | 시험 시나리오·데이터·환경, 예외상황·구현기능 완전성 확인 |

### 2.2 점검항목 / 검토항목 체크리스트

- [ ] **`SD22-2-A1`** 개발대상 시스템의 업무영역이 충분히 분석되었는가? (업무 식별·우선순위, 수행자-업무 상호작용, 업무흐름, 법제도/위험요인)
- [ ] **`SD22-2-B1`** 누락 기능 없이 유스케이스 모형을 정제·실체화하였는가? (정의·역할·액터 누락, 시나리오·이벤트흐름, 변경/확장/유지보수 용이성)
- [ ] **`SD22-2-C1`** UI 프로토타이핑이 계획대로 실시되고 결과가 검토/반영되었는가?
- [ ] **`SD22-2-D1`** 필요한 UI를 명확히 식별하고 흐름을 적절히 설계하였는가?
- [ ] **`SD22-2-D2`** UI·보고서 양식을 상세하고 일관성 있게 설계하였는가?
- [ ] **`SD22-2-E1`** 유스케이스로부터 분석클래스가 충분히 도출·상세화되었는가?
- [ ] **`SD22-2-E2`** 설계된 클래스가 구현 가능한 수준으로 상세화되었는가? (추적성, 객체 식별, 속성/연산/역할 명세, 구현언어 반영)
- [ ] **`SD22-2-F1`** 내/외부 시스템 연계 분석/설계가 이루어졌는가? (대상·데이터 식별, 기술방식, 주기·방법)
- [ ] **`SD22-2-G1`** 사용자 접근통제 및 보안 분석이 설계에 반영되었는가?
- [ ] **`SD22-2-H1`** 컴포넌트 내부구조를 정제하였는가? (클래스/인터페이스 도출, 지속성, 개발 가능 수준)
- [ ] **`SD22-2-H2`** 공통성·가변성 분석이 충분한가? (재사용 선정기준·후보, 차이 분석, 공통컴포넌트 식별)
- [ ] **`SD22-2-H3`** 구현/도입할 컴포넌트를 구체적으로 명세하였는가?
- [ ] **`SD22-2-H4`** 시스템에 적합한 컴포넌트를 도입하였는가?
- [ ] **`SD22-2-H5`** 컴포넌트 아키텍처를 정의하고 적절히 구조화하였는가?
- [ ] **`SD22-2-I1`** 단위시험 계획이 적정하게 수립되었는가? (시나리오·데이터·환경, 예외/완전성 확인)

## 3단계 · 구현 (SD23-2)

> **개요**: 설계에 따라 응용시스템의 기능 완전성·무결성·편의성·적정성을 확보하도록 컴포넌트 도입/개발로 구현하고, 단위기능 검증을 수행하였는지 점검.

### 3.1 필수 산출물 및 작성기준

| # | 산출물 | 작성기준 |
|---|--------|----------|
| 1 | **컴포넌트 코드** | 요구사항/유스케이스 대비 기능 구현 충분성·완전성, 업무흐름에 따른 기능흐름 |
| 2 | **구현된 응용시스템** | 사용자 이해·학습 용이성·사용 편의성, 사용자 오류 방지 |
| 3 | **단위시험 계획/결과서** | 컴포넌트 개별기능 완전성, 예외상황 시나리오, 입출력 데이터 정확성, 오류 개선·결과 관리 |
| 4 | **통합시험 계획서** | 컴포넌트 간 인터페이스 완전성, 시험환경·절차·시나리오·데이터 |

### 3.2 점검항목 / 검토항목 체크리스트

- [ ] **`SD23-2-A1`** 분석/설계된 모든 기능이 적정하게 구현되었는가? (충분성, 업무흐름 기능흐름, 완전성)
- [ ] **`SD23-2-B1`** 구현 기능·화면이 사용자 편의성을 충분히 확보하였는가? (이해·학습·사용 편의성, 오류 방지)
- [ ] **`SD23-2-C1`** 내/외부 시스템 인터페이스가 정확하게 구현되었는가? (연계 기능·데이터·주기·방식)
- [ ] **`SD23-2-D1`** 사용자 접근통제 및 보안사항이 적정하게 구현되었는가? (접근권한·감사기능, 보안기술 적용)
- [ ] **`SD23-2-E1`** 단위시험이 계획대로 실시되고 결과 평가에 따른 개선이 이루어졌는가? (개별기능 완전성, 예외 시나리오, 입출력 정확성, 오류 개선·관리)
- [ ] **`SD23-2-F1`** 통합시험 계획이 수립되었는가? (컴포넌트 간 인터페이스 완전성, 환경·절차·시나리오·데이터)

## 4단계 · 시험활동 (SD14-1)

> **개요**: 통합시험·시스템시험을 통해 구현된 시스템이 통합적 관점에서 기능 완전성·성능·안전성·보안성 확보 여부를 검증하였는지 점검.

### 4.1 필수 산출물 및 작성기준

| # | 산출물 | 작성기준 |
|---|--------|----------|
| 1 | **시험계획서** | 시험대상·전제사항·품질목표, 시험 데이터/응용시스템/기반환경 준비현황 |
| 2 | **통합시험 시나리오·결과서** | 시험범위 적정성, 결과 정확성, 절차 준수, 내·외부 연계/통합 무결성 |
| 3 | **시스템시험 결과서** | 부하수준 적정성, 목표 성능·가용성·보안성 만족 여부 |
| 4 | **시험결과 관리대장(오류/결함 관리)** | 시험 유형별 시나리오·데이터·결과 관리, 오류 개선 이력 |
| 5 | **시스템 최적화(튜닝) 보고서** | 튜닝 대상선정·방법·일정, 파라미터/응용 최적화, 튜닝 전후 성능평가 |
| 6 | **사용자/운영자 지침서** | 장비관리·백업·복구, 시스템 구성도·설치·환경설정, 사용방법, 현행화, 교육·편의성 |
| 7 | **사용자 인수시험 계획서** | 사용자 시험환경·절차·시나리오·데이터 |

### 4.2 점검항목 / 검토항목 체크리스트

- [ ] **`SD14-1-A1`** 시험환경이 실제 운영환경을 반영해 구축되고 준비가 완료되었는가?
- [ ] **`SD14-1-B1`** 통합시험이 계획대로 적정하게 실시되었는가? (범위·결과·절차)
- [ ] **`SD14-1-B2`** 통합시험을 통해 연계 및 기능 완전성·데이터 무결성이 확보되었는가?

## 부록 A. 전 단계 공통 — 요구사항 추적성(RTM) 핵심 점검

감리에서 가장 빈번하게 지적되는 항목이 **추적성 단절**입니다. 다음 추적 체인이 끊기지 않아야 합니다.

```
요구사항(REQ) → 유스케이스(UC) → 분석/설계클래스(CLS) → 컴포넌트/코드(CMP) → 단위시험(UT) → 통합/인수시험(IT/UAT)
```

- [ ] 모든 요구사항 ID가 유스케이스에 매핑됨 (고아 요구사항 없음)
- [ ] 모든 유스케이스가 설계 클래스/화면에 매핑됨
- [ ] 모든 설계 항목이 구현 코드/컴포넌트에 매핑됨
- [ ] 모든 구현 기능이 단위시험 케이스에 매핑됨
- [ ] 모든 요구사항이 인수시험 케이스로 검증됨
